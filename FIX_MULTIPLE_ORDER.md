# Fix: Multiple Orders per Ticker

## Observed behaviour

Running the pipeline with `--once` against NTIC produced **8 orders for the same ticker in a single cycle**, with mixed LONG and SHORT directions. Alpaca rejected 3 of them as "potential wash trade" (opposing-side orders on the same ticker). 5 went through.

---

## Root cause — two compounding problems

### Problem 1 — Stale multi-quarter articles in the RSS feed

`NewsIngestorAgent` ingested 10 NTIC headlines in a single cycle spanning **three different earnings quarters** (Q4 2025, Q1 2026, Q2 2026). These are not about the same event. Every article is treated as a fresh, independent signal regardless of when it was published.

### Problem 2 — 1 news article → 1 sentiment → 1 signal → 1 order (no per-ticker consolidation)

`SignalGeneratorAgent.run()` iterates `sentiment_results` one-by-one (`signal_generator.py:170`). With 10 input articles it emits up to 10 signals for the same ticker — some LONG (bullish articles), some SHORT (bearish articles). The pipeline has **no consolidation step** between sentiment results and order submission.

`RiskManagerAgent` has a within-batch ticker dedup check (L3a: reject if ticker already approved in the same batch), but `RISK_DRY_RUN=True` force-approves every signal regardless of any check. In production (dry_run=False) only the *first* signal per ticker per cycle would survive L3a — but which one wins is non-deterministic because `SentimentAnalystAgent` runs the LLM calls concurrently.

### Execution flow that triggered the behaviour

```
NewsIngestorAgent       → 10 NTIC articles (Q4 2025, Q1 2026, Q2 2026)
SentimentAnalystAgent   → 10 SentimentResult objects (mixed: 5 bullish, 2 bearish, 1 very_bearish, 2 neutral)
SignalGeneratorAgent    → 8 TradeSignal objects (2 neutral dropped; signals are LONG and SHORT)
RiskManagerAgent        → all 8 force-approved (RISK_DRY_RUN=True bypasses every check)
ExecutionAgent          → 8 order submissions for NTIC
                          → 3 rejected by Alpaca (wash trade — opposing sides)
                          → 5 accepted
```

---

## Are these EARN_PRE allocations?

**No.** These are raw `type=earnings` events sourced from the RSS feed via `NewsIngestorAgent`. EARN_PRE events are generated by `EarningsTickerNode`, which reads from the DB populated by `EarningsCalendarAgent` (cron at 07:00 ET). The run happened at 22:55 before any cron-populated NTIC entries existed in the DB, so no EARN_PRE was involved.

---

## Is it conceptually correct to have multiple orders per ticker?

**No.** The current model is incoherent for two reasons:

1. **Contradictory directions in the same cycle** — LONG + SHORT orders for the same ticker is meaningless and triggers broker wash-trade rejection.
2. **Ordering lottery** — in production (dry_run=False), L3a dedup passes the *first* approved signal per ticker per cycle. Which one wins depends on LLM call return order, which is non-deterministic.

---

## Recommended fix — two steps, no new LLM calls

### Step 1 — Recency filter in `NewsIngestorAgent` (highest impact, cheapest)

Add a `MAX_ARTICLE_AGE_HOURS` config setting (e.g. 48 h default). Filter out articles whose publish date is older than the cutoff before they enter the pipeline. This eliminates stale multi-quarter articles at the source. The NTIC case would have gone from 10 events to 2–3 current ones.

### Step 2 — Deterministic per-ticker aggregation in `SignalGeneratorAgent`

Instead of iterating `sentiment_results` one-by-one, group by ticker first and produce **one synthetic `SentimentResult` per ticker** using a confidence-weighted average:

```python
# group sentiment_results by ticker
for ticker, group in grouped_by_ticker:
    directions = {r.label for r in group if r.label not in NEUTRAL_LABELS}
    if len(directions) > 1:
        # articles disagree on direction → skip this ticker (MIXED)
        continue
    total_weight = sum(r.confidence for r in group)
    score_agg = sum(r.score * r.confidence for r in group) / total_weight
    confidence_agg = mean(r.confidence for r in group)
    # build one synthetic SentimentResult → one signal → one order
```

- Zero extra LLM calls.
- Fully deterministic and auditable.
- Consistent with the Pattern C philosophy (`ConfidenceScorer` is also intentionally deterministic).

## Options considered and rejected

### Option A — Aggregating LLM after N sentiment results

Feed pre-scored `SentimentResult` objects to an LLM, get one consolidated verdict per ticker.

**Rejected:** aggregating LLM outputs via another LLM amplifies noise rather than reducing it. The existing individual scores already carry uncertainty; a second LLM step adds cost, latency, and non-determinism without resolving the stale-article problem. Each call also needs publish dates passed explicitly for the LLM to reason about recency.

### Option B — Single LLM call per ticker with all raw article text

Skip per-article sentiment scoring entirely. Feed all raw headlines/summaries for a ticker into one prompt, get one direction + conviction + rationale.

**Stronger than Option A** — a single well-prompted LLM can identify the most recent event, discard stale quarters, and reason holistically. The genuine tradeoff is larger prompts (grows linearly with article count) and loss of per-article audit trail.

**Not recommended as the first fix** — Steps 1+2 above solve the problem deterministically. Option B becomes worthwhile only if mixed-quarter noise persists after the recency filter is in place, or if per-article audit trail is not a requirement. It would also require merging `SentimentAnalystAgent` and `SignalGeneratorAgent` into a single node, which is a larger structural change.
